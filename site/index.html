The Journey

How a product leader and an AI built this together

This wasn't a spec handed to engineering. It was a live conversation — exploring, iterating, and shipping. Here's how it unfolded.

Phase 1 — The Spark

Learning what MCP even is

Started with a goal: understand MCP by building one. We entered planning mode — Claude explained the protocol, the architecture, and the mental model. Then we scaffolded a minimal server with one tool (query NYC crime data from Socrata).

Let's take on a new project. I want to understand exactly what it means and what it takes to build an MCP server, and build one myself.
Phase 2 — First Working Tool

Hello world: querying NYC crime data

Connected the server to Claude Desktop. First query: "What are recent crime incidents in NYC?" — Claude called the MCP tool, hit the Socrata API, and returned real data. The mental model clicked.

Phase 3 — Expanding the Vision

From 1 city to 5, from 1 source to 8

Added Census demographics (any US city), FRED economics, BLS employment, FBI crime stats. Each source followed the same pattern — learn the API, write a client, register a tool. The conversation shifted from "how does this work" to "what should we add next."

Are there other data sources we can add? I want this to be useful for real city comparison.
Phase 4 — The Cohort Builder

Finding peer cities by similarity

Built a tool that finds cities similar to any target across multiple dimensions. Then expanded it to pull from ALL data sources — demographics, economics, crime, employment. Two versions: fast (Census-only) and rich (multi-source).

Maybe a 'create cohort' tool? This tool suggests a list of peer cities based on real data.
Phase 5 — Deploy + Share

Live on Railway in minutes

Deployed to Railway with auto-deploy from GitHub. Added HTTP transport mode alongside stdio. Shared with the team via Slack — "look what we can do."

Phase 6 — The Community Voices Bridge

From hard data to resident experience

Asked: "What if we could cross-reference this with community voices data?" This led to 311 trends, transit, schools, permits, and city budgets — the data types that map directly to what residents care about. Then built three analysis tools that bridge the gap.

What else can we think of? Especially if we're thinking how to cross-reference these data points with community voices data.
Phase 7 — Dynamic Resolution

Making it work for ANY city

The big unlock: a shared geo-resolver that turns any city name into coordinates + FIPS codes + ZIP. Now 10 of 15 sources work for any US city — not just the 50 we hardcoded. Critical for Zencity's mid-size city clients.

Is it possible to do dynamic pulls for specific cities that aren't in the included list? Most of Zencity's clients aren't on these lists.
What We Learned

Working with AI is like working with a developer — but different

The dynamics of the collaboration mattered as much as the output.

01

Start with the question, not the spec

We didn't write a PRD. We said "I want to understand MCP" and built from there. The spec emerged from the conversation, not the other way around.

02

Iterate in conversation

Every "what if we also..." moment expanded the scope naturally. The cohort builder, the geo-resolver, the analysis tools — all emerged from follow-up questions, not upfront planning.

03

Think in layers

First one source, then many sources, then analysis on top, then dynamic resolution underneath. Each layer made the next one possible. You don't need to see the whole architecture upfront.

04

Push on coverage, not just features

"Can we expand the city list?" was the question that made this actually useful. Features without coverage are demos. Coverage without features is a database.

05

Deploy early, share fast

We deployed to Railway mid-build and shared on Slack immediately. The feedback loop matters more than polish.

06

Let the product brain lead

The AI knows APIs and code patterns. You know what's valuable. "Cross-reference with community voices" was a product insight that shaped the entire second half of the build.

Try It

Connect in 5 minutes

Two ways to get city data flowing into Claude. Pick whichever you use.

1

Clone & build

git clone https://github.com/zencity-product/city-data-mcp.git cd city-data-mcp npm install && npm run build
2

Get API keys

Two required keys (free, instant signup). The rest are optional and unlock more data sources.

Census API RequiredDemographics, permits, housing
Get key →
FRED API RequiredEconomics, unemployment, housing index
Get key →
AirNow API OptionalAir quality index & forecasts
Get key →
Google Civic OptionalElected representatives
Get key →

UK sources — 10 of 13 UK tools work without keys. Optional keys for weather, transport, and representatives:

Met Office OptionalUK weather & forecast
Get key →
TfL API OptionalLondon transport status
Get key →
TheyWorkForYou OptionalUK MPs & councillors
Get key →
3

Add to Claude Desktop config

Open Settings → Developer → Edit Config in Claude Desktop, then paste:

{ "mcpServers": { "city-data-mcp": { "command": "node", "args": ["/FULL/PATH/TO/city-data-mcp/dist/index.js"], "env": { "CENSUS_API_KEY": "your-key-here", "FRED_API_KEY": "your-key-here" } } } }

Replace /FULL/PATH/TO/ with where you cloned the repo.

4

Restart Claude & go

Quit and reopen Claude Desktop. You should see the city-data-mcp tools in the hammer icon. Try:

"Compare demographics for Denver, Austin, and Portland" "Create a full city briefing for San Antonio" "What are the economic trends for the Seattle metro?"

Or add it to your project's .mcp.json:

{ "mcpServers": { "city-data-mcp": { "command": "node", "args": ["/FULL/PATH/TO/city-data-mcp/dist/index.js"], "env": { "CENSUS_API_KEY": "your-key-here", "FRED_API_KEY": "your-key-here" } } } }
4

Start using it

The tools are immediately available. Just ask Claude:

"Create a city briefing for Memphis" "Find peer cities for Boulder based on demographics and economics" "What's the air quality and weather in Phoenix right now?"
Already deployed remotely? The server also supports HTTP mode for shared access. Run with --http flag or set the PORT env var. See the README for details.
